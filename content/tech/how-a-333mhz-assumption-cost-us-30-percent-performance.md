## Situation (情境背景)

在進行 Zynq UltraScale+ MPSoC 7EV 的 HEVC 高負載影像編碼串流測試時，系統面臨嚴重的效能瓶頸。

預期表現： 根據晶片規格與硬體算力，該 VCU 影像流水線應能輕鬆達到 86 fps 的極限吞吐量。

實際表象： 系統幀率被死死卡在 58 fps，效能平白損失了約 30%。即使軟體端完全不填入實體資料、只跑空訊號，幀率依然毫無起色，排除軟體層面記憶體塞車的可能。



## Task (面臨任務)

我必須主導跨軟硬體邊界（FPGA PL 與 Linux PS）的 Root-Cause 分析，找出隱藏在晶片底層的無形氣阻（Stall），並在不更動硬體演算法與算力架構的前提下，將 VCU 效能滿血恢復至 86 fps。



## Action (核心行動：直擊自動化盲點的偵探歷程)
### Phase A: 建立宏觀物理模型（高速公路理論）

我意識到這不是一個「頻率不夠快」的優化問題，而是一個「速度不均勻引發集體塞車」的系統震盪問題。
這就像三線道高速公路上，如果大家速限一致（60 km/h），車流會很穩定；但若有人忽快忽慢或頻繁踩急煞（130 km/h 瞬間減速到 20 km/h），後方流水線就會引發雪崩式的連鎖回堵。我開始尋找晶片內部的「那個急煞車訊號」。

### Phase B: 審查時脈樹與工具鏈，破除「自動化代碼」盲點
我與 FPGA 同事深入 Vivado 的硬體配置，並交叉審查 Linux Kernel 的 Common Clock Framework (CCF) 與生成的裝置樹（Device Tree），揪出了隱藏在工具鏈中的骨牌效應：

實體硬體現實： 電路板（PCB）上的實體晶振，物理上灌入的是 33.3333 MHz 的方波。

源頭的硬體誤填： 經追查，FPGA 工程師先前在 Vivado 配置 VCU IP 屬性（Property）時，將 VCU_REF_CLK 誤填成了整數的 33000000 (33.00 MHz)。

PetaLinux 自動化背鍋： 當硬體編譯完成並導出 .xsa 檔後，PetaLinux 的 Device Tree Generator (DTG) 在自動建構系統時，盲目地信賴了硬體導出的數據，自動生成了帶有 clock-frequency = <33000000>; 的 Device Tree 節點。

驅動程式被欺騙： 由於這是一段由系統自動生成的代碼（Auto-generated code），Linux 驅動工程師在正常狀況下不會去懷疑它的正確性。Linux VCU 核心驅動就這樣讀取了這個錯誤的裝置樹參數，用 33MHz 的假基底去計算 PLL 的除頻與倍頻公式。

### Phase C: 揭開「消失的第 100 齒」硬體連鎖效應
我提出了齒輪錯位假設：當外部實體是 100 齒（33.33MHz），內部 PLL 設定卻被自動化軟體公式帶入 99 齒（33MHz）時，每到第 100 步，時域相位就會產生 1.0% 的強烈撞擊與非同步錯位：

1. 觸發 CDC 亞穩態保護： 晶片硬體為了安全，強制插入 2-3 個時脈週期的等待（Wait State）來穩定訊號。

2. 流水線回堵（Pipeline Backpressure）： VCU 內部的深層影像流水線被迫集體急煞，重新填滿資料需要額外耗費多個週期。

3. AXI 突發傳輸（Burst）中斷： 正在連續搬運的 NV16 巨量資料包被打斷，系統被迫重新發起總線仲裁與 Protocol 握手。

這 1% 的源頭誤差在幾何放大下，演變成宏觀視角下 30% 的效能雪崩。



## Result (最終成果)

精準修正： 我們將 Vivado 的硬體屬性修正為正統的 33333333 (33.33 MHz)。重新導出後，PetaLinux 自動生成了正確的 Device Tree，驅動程式算出的 PLL 倍頻係數與實體硬體完美嚙合（100 齒對齊 100 齒）。

效能暴衝 30%： 封印徹底解除，晶片內部再無任何保護性急煞與流水線回堵，編碼吞吐量從 58 fps 瞬間噴發至 86 fps，達到硬體極限。

可觀測性沉澱： 將此「跨時域、跨硬體/工具鏈/軟體」的除錯經驗沉澱為團隊的 CI/CD 檢查機制，避免後續專案再度因過度信任自動化工具而踩坑。
